# 外部結合テスト（ITb）シナリオ定義

**対象**: WBS No.8 外部結合テスト（ITb）  
**目的**: 画面（FE）とロジック（BE）の疎通を、実務フローに沿ったエンドツーエンドのシナリオで検証する。  
**参照**: [検収環境整備ガイド](TEST_GUIDE.md)、[デバイス・マトリクス](apps/qr-attendance/docs/FE_DEVICE_MATRIX_AND_HYBRID_UI.md)

---

## 1. シナリオ一覧

| ID | 概要 | 主な役割 | 合格の目安 |
|----|------|----------|------------|
| **シナリオ1** | 生徒の新規登録〜ログイン〜お知らせ確認〜イベント申込 | 管理者 → 生徒（スマホ） | 申込がDB・画面に反映され、一覧で確認できること |
| **シナリオ2** | 当日QR打刻〜スタッフの出席確認〜管理者のCSV出力 | 生徒 → スタッフ → 管理者 | 打刻が出席一覧に表示され、CSVに含まれること |

---

## 2. シナリオ1: 生徒の新規登録 → ログイン → お知らせ確認 → イベント申込

### 2.1 フロー概要

1. **管理者**がPCで生徒を新規登録する。
2. **生徒**がスマホでログインする。
3. **生徒**がホームでお知らせを確認する。
4. **生徒**がイベントに申し込む。

### 2.2 操作手順と確認ポイント

| ステップ | 操作者 | 操作内容 | 確認内容（合格基準） |
|----------|--------|----------|----------------------|
| 1-1 | 管理者 | 管理者でログインし、生徒名簿管理（/admin/students）を開く。 | 一覧が表示され、新規登録が可能であること。 |
| 1-2 | 管理者 | 新規登録でメール・パスワード・氏名（漢字・カナ）等を入力し保存。 | 成功メッセージが出て、一覧に当該生徒が表示されること。 |
| 2-1 | 生徒 | **スマホ**（またはスマホ幅のブラウザ）でログイン画面を開き、1-2で登録したメール・パスワードでログイン。 | ログイン成功し、利用者用ホーム（/home）に遷移すること。 |
| 2-2 | 生徒 | ホーム画面で「重要なお知らせ」ブロックを確認。 | 公開・掲載期間内のお知らせが1件以上あればブロックが表示され、重要は赤バッジ/枠で表示されること。0件の場合はブロック非表示でよい。 |
| 3-1 | 生徒 | イベント一覧（/events）を開き、公開中のイベントを1件選ぶ。 | イベント一覧が表示され、詳細・申込が可能であること。 |
| 3-2 | 生徒 | 当該イベントの「申し込む」を実行。 | 申込成功のフィードバックがあり、参加申込一覧（/user/registrations）に当該イベントが表示されること。 |
| 3-3 | 管理者 | 参加申込一覧（全利用者）（/admin/registrations）を開く。 | 当該生徒の当該イベントへの申込が1行で表示され、「出席確認」リンクで該当イベントの出席確認画面へ遷移できること。 |

### 2.3 実行計画

| 項目 | 内容 |
|------|------|
| **実行者** | 1名で役割を切り替えても可。または管理者役・生徒役を分けても可。 |
| **環境** | フロント（port 3000）・バックエンド（port 3001）・DB が起動した状態。生徒操作は実機スマホまたは DevTools で 767px 幅にしたブラウザで実施。 |
| **合格判定** | 上記「確認内容（合格基準）」をすべて満たした場合にシナリオ1を**合格**とする。 |
| **不具合時** | 失敗したステップ、エラー表示・ネットワーク応答（4xx/5xx）を記録し、原因を切り分けする。 |

## 3. シナリオ2: 当日QR打刻 → スタッフの出席確認 → 管理者のCSV出力

### 3.1 フロー概要

1. **生徒**が当日、打刻用QRを表示してスタッフにスキャンしてもらう（または手動打刻で代替）。
2. **スタッフ**が出席確認画面でリアルタイムに打刻を確認する。
3. **管理者**が当該イベントの出席レポートをCSV出力する。

### 3.2 操作手順と確認ポイント

| ステップ | 操作者 | 操作内容 | 確認内容（合格基準） |
|----------|--------|----------|----------------------|
| 1-1 | 生徒 | シナリオ1で申し込んだイベントの「当日」に、打刻用QR表示（/my-qr）を開く。 | 有効なQRが表示されること。 |
| 1-2 | スタッフ | 打刻スキャン（/staff/scan）で当該生徒のQRをスキャンする。または手動打刻（/staff/manual）で当該生徒を検索し、当該イベントを選んで打刻。 | 打刻成功のフィードバックがあること。 |
| 2-1 | スタッフ | 出席確認（参加者一覧）（/admin/events/[eventId]/participants）を開く。 | 当該イベントの参加者一覧が表示され、当該生徒の行に**打刻済み**（入室時刻等）が表示されること。 |
| 2-2 | スタッフ | 一覧の並び・フィルタや、PC/スマホ表示切替（768px境界）を任意で確認。 | 767px でカード＋警告バナー、768px でテーブル表示となること。 |
| 3-1 | 管理者 | 当該イベントの出席レポート（/admin/events/[eventId]/attendance-report）を開く。 | サマリー（申込数・出席数・出席率）に当該生徒の打刻が反映されていること。 |
| 3-2 | 管理者 | 「CSVをダウンロード」を実行。 | 日本語BOM付きUTF-8のCSVがダウンロードされ、ファイル名が「出席レポート_イベント名_出力日.csv」形式であること。 |
| 3-3 | 管理者 | ダウンロードしたCSVを開く。 | 当該生徒の行が含まれ、入室時刻（および必要に応じて退室時刻）・出席率等が正しく記載されていること。 |

### 3.3 実行計画

| 項目 | 内容 |
|------|------|
| **実行者** | 1名で生徒・スタッフ・管理者の順に役割を切り替えても可。または3役を分けても可。 |
| **前提** | シナリオ1が完了し、当該生徒が当該イベントに申込済みであること。イベントの開催日が「当日」であること（または手動打刻で時刻を補う）。 |
| **環境** | フロント・バックエンド・DB が起動した状態。スタッフ・管理者操作はPC/タブレットを想定。生徒のQR表示はスマホ想定。 |
| **合格判定** | 上記「確認内容（合格基準）」をすべて満たした場合にシナリオ2を**合格**とする。 |
| **二重打刻防止の確認** | 打刻（QRスキャンまたは手動打刻）を**同一生徒・同一イベントで連打**し、参加者一覧・出席レポート・CSV に**1人1件のみ**表示されること。2回目以降の打刻APIは「入室打刻は既に記録済みです」等で成功応答し、新規レコードは増えないこと。 |
| **不具合時** | 失敗したステップ、API の応答・CSV の内容を記録し、BE/FE/DB のどれで不整合が起きているかを切り分ける。 |

## 4. 実行計画サマリ（誰が・何を・合格基準）

| シナリオ | 主な操作者（役割） | 合格とする基準 |
|----------|--------------------|----------------|
| **シナリオ1** | 管理者（生徒登録）→ 生徒（スマホでログイン・お知らせ・申込）→ 管理者（申込一覧で確認） | ① 生徒が新規登録でき、② そのアカウントでスマホログインでき、③ ホームでお知らせが表示され（条件を満たす場合）、④ イベント申込が成功し、⑤ 管理者の参加申込一覧にその申込が表示され、出席確認リンクで遷移できること。 |
| **シナリオ2** | 生徒（QR表示）→ スタッフ（打刻・出席確認）→ 管理者（レポート・CSV） | ① 生徒の打刻が成功し、② スタッフの出席確認画面に打刻がリアルタイムで表示され、③ 管理者の出席レポートに反映され、④ CSV に当該生徒の打刻データが含まれていること。 |

---

## 5. テスト実施時の記録様式（案）

実施日・実施者・環境（FE/BE の URL・DB 名等）を記録し、各ステップの結果を「合格 / 不合格」で記録する。  
不合格の場合は、エラー内容・スクリーンショット・ネットワークログ等を添付する。

| 2 | 1-1 〜 3-3 | 合格 / 不合格 | |

## 6. 次のアクション

- 上記シナリオ案をレビューし、必要に応じて文言・ステップを調整する。
- 調整後、テスト実施者が `TEST_GUIDE.md` に沿って環境を整え、シナリオ1・2を実行する。
- 両シナリオが合格した時点で、WBS No.8 を**完了**に更新する。

以上を **外部結合テスト（ITb）のシナリオ案** とする。
