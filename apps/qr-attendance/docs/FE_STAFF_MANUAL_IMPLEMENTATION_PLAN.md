# 手動打刻画面（/staff/manual）実装案 — QR優先思想に基づく

**前提**: [FE_UI_UX_DESIGN_2-1.md](./FE_UI_UX_DESIGN_2-1.md) §0.1（主従関係・QR優先）、§2.3、§5 および [FE_2-3_HOME_MENU_DESIGN.md](./FE_2-3_HOME_MENU_DESIGN.md) §0 を満たす。  
**位置づけ**: 手動打刻は **QRが利用できない場合の代替措置（レスキュー・モード）**。受付のメインは **/staff/scan（QRスキャン）**。本画面は **スキャン画面から呼び出されるサブ画面** とする。

---

## 1. 業務上の原則（PM確定・遵守必須）

| 原則 | 内容 |
|------|------|
| **主従関係** | 本システムのデフォルトはQRコード打刻。手動打刻は**例外フロー**に過ぎない。 |
| **導線の主役** | 受付担当者の**メイン画面は /staff/scan**。手動打刻は**そこから呼び出されるサブ画面**。 |
| **完了後の強制リセット** | 手動打刻完了後は **確認メッセージ → 「QRスキャン画面に戻る」ボタンを強調**。**自動で次の生徒検索へ進むループは認めない。** |

---

## 2. 画面構成・導線

### 2.1 入口

| 優先度 | 入口 | 説明 |
|--------|------|------|
| **主** | `/staff/scan`（QRスキャン画面） | 画面上に「QRが読めない場合 → 手動打刻へ」リンク／ボタンを配置し、`/staff/manual` へ遷移させる。 |
| **副** | `/home` | スタッフ以上のホームメニューに「手動打刻（代替・レスキュー）」としてリンクを出してもよい。 |

- **スキャン画面側の対応**: `/staff/scan` に「QRが使えない場合は手動打刻」リンクを必ず設け、手動打刻の**主な入口**をスキャン画面にする。

### 2.2 画面内フロー（1回の打刻単位）

1. **イベント選択**（必須）  
   - ドロップダウンまたは一覧でイベントを選択（GET /v1/admin/events または GET /v1/events 等で取得）。
2. **生徒検索**  
   - テキスト入力（漢字・ふりがな）→ GET /v1/students/search?q=...（最小2文字程度）。登録済み生徒のみヒット。
3. **打刻実行**  
   - 検索結果から1名選択 → POST /v1/attendance/manual（event_id, email 等）。  
   - 成功: **「〇〇さんを打刻しました」等の確認メッセージを表示**。  
   - 409: 「既に打刻済みです」表示。
4. **完了後の必須挙動（強制リセット）**  
   - 打刻成功後は **同じ画面で「次の生徒を検索」に自動で進まない**。  
   - **確認メッセージの下に「QRスキャン画面に戻る」ボタンを強調表示**（プライマリCTA）。クリックで `/staff/scan` へ遷移。  
   - オプションで「もう1件手動打刻する」リンクを小さく出してもよいが、**デフォルトの導線は「QRスキャンに戻る」** とする。

### 2.3 禁止する挙動

- 打刻完了後に **自動で検索フォームをクリアして「次の生徒を検索」状態にし、手動打刻をメインとするループ** にすることは**認めない**。  
- 複数人連続で手動打刻する場合でも、**毎回「QRスキャン画面に戻る」を選べる** ことを明確にし、**主導線はQRスキャンに戻す** ことを徹底する。

## 3. UI要件（詳細）

### 3.1 レイアウト

- **ヘッダー**: 「手動打刻（QRが使えない場合の代替）」等のタイトル + **「QRスキャン画面に戻る」リンク**（常時表示で、打刻完了後も同じリンクを強調して再表示）。
- **メイン**  
  - イベント選択 UI  
  - 生徒検索入力 + 検索ボタン  
  - 検索結果一覧（name_kanji, name_kana, email + 「打刻する」等のアクション）  
  - 打刻実行後: **確認メッセージ + 大きく目立つ「QRスキャン画面に戻る」ボタン**

### 3.2 確認メッセージ・完了ブロック

- 打刻成功時:  
  - 例: 「**〇〇 さんを打刻しました。**」  
  - その直下に **プライマリボタン「QRスキャン画面に戻る」**（href="/staff/scan"、視覚的に強調）。  
  - 必要ならその下に小さく「もう1件手動で打刻する」リンク（クリックで検索フォームを表示し直す等）。  
- 打刻失敗（409）時:  
  - 「既に打刻済みです。」等のメッセージ + 同様に **「QRスキャン画面に戻る」** を表示。

### 3.3 役割別表示（既存方針の維持）

- **スタッフ（Role: 2）**: 検索・打刻のみ。新規登録の入口は表示しない。検索でヒットしない場合は「該当する生徒がいません。名簿にない場合は管理者にご依頼ください」等。
- **管理者（Role: 3）**: 上記に加え、「名前で見つからない場合 → 生徒名簿で新規登録」リンク（`/admin/students`）。登録後は手動打刻へ戻れるようにする。**いずれの場合も打刻完了後は「QRスキャン画面に戻る」を強調。**

### 3.4 API

- イベント一覧: GET /v1/admin/events または GET /v1/events（スタッフ以上で叩けるAPI）  
- 生徒検索: GET /v1/students/search?q=...（スタッフ以上）  
- 手動打刻: POST /v1/attendance/manual（body: event_id, email または user_id）（スタッフ以上）
