# タスク 2-3. ホーム画面・役割別メニュー表示 — 設計案

**WBS**: 実装 (Coding) - FE（2-3 対応中）  
**タスク**: 2-3. ホーム画面・役割別メニュー表示  
**前提**: [SECURITY_DESIGN.md](./SECURITY_DESIGN.md) の権限マトリックスに基づき、ログイン済みユーザーの役割（利用者 / スタッフ / 管理者）に応じて表示するメニューを動的に出し分ける。

## 0. 受付まわりの主従関係（QR優先思想・PM確定）

| 方針 | 内容 |
|------|------|
| **デフォルトはQR打刻** | 本システムの受付の**主軸**は **QRコード打刻** である。 |
| **手動打刻は例外フロー** | 手動打刻（`/staff/manual`）は、**QRが利用できない場合の代替措置（レスキュー・モード）** に過ぎない。 |
| **受付のメイン画面** | 受付担当者の**メイン画面は `/staff/scan`（QRスキャン）** とする。 |
| **手動打刻の位置づけ** | 手動打刻画面は **スキャン画面から呼び出されるサブ画面** とする。導線の主役はQRスキャンであり、手動打刻完了後は必ず **QRスキャン画面に戻る** ことを徹底する（手動打刻をメインとするループは認めない）。 |

- ホームから「手動打刻」へのリンクは、スタッフ以上に**代替入口**として出してもよいが、**メインの導線は /staff/scan → （必要時）手動打刻 → /staff/scan に戻る** とする。

## 1. 役割定義（権限マトリックスとの対応）

| role_flag | 役割名   | 想定ユーザー   | ホームで表示するメニュー |
|-----------|----------|----------------|---------------------------|
| 1         | 利用者   | 参加者・生徒   | 認証共通 + **利用者専用**のみ |
| 2         | スタッフ | 受付・運営     | 認証共通 + 利用者専用 + **スタッフ専用** |
| 3         | 管理者   | システム管理者 | **全メニュー**（管理者専用を含む） |

- 表示可否は **AuthContext の `roleFlag`（= API ログイン応答・localStorage の値）** に基づく。`useAuth()` の `isAdmin`, `isStaff`, `role`, `roleLabel` で判定する。
- バックエンドは SECURITY_DESIGN の通り API 単位で 401/403 を返すため、**フロントの出し分けは UX のため**であり、権限の正は API 側にある。

## 2. 役割別メニュー一覧（表示するカード・リンク）

### 2.1 全役割で表示（認証共通・利用者向け機能）

| 表示ラベル       | パス              | 説明                         | 対応API・備考 |
|------------------|-------------------|------------------------------|---------------|
| 打刻用QR表示     | `/my-qr`          | 受付でスタッフにスキャンしてもらうQR | GET /v1/users/me/qr |
| イベント一覧     | `/events`         | 公開イベント閲覧・申込       | GET /v1/events |
| 打刻履歴         | `/history`        | 自分の打刻履歴               | GET /v1/users/attendance/history |
| 参加申込一覧     | `/registrations` または `/user/registrations` | 自分が申し込んだイベント | GET /v1/users/registrations |
| マイページ       | `/user/me`        | 自分のプロフィール           | （任意） |

※ パスは既存ルート（`/user/registrations`, `/user/events`, `/history`, `/my-qr` 等）と 2-1 の案のどちらかに一本化してよい。本設計では「表示するメニュー」と「パス」の対応だけ定義する。

### 2.2 スタッフ・管理者のみ表示（スタッフ専用）

| 表示ラベル           | パス              | 説明                               | 対応API・備考 |
|----------------------|-------------------|------------------------------------|---------------|
| **打刻スキャン（スタッフ用）** | `/staff/scan` または `/staff/attendance` | **受付のメイン画面**。利用者QRをスキャンして打刻。QRが使えない場合のみ手動打刻へ。 | POST /v1/users/attendance |
| 手動打刻（代替・レスキュー） | `/staff/manual`   | **QRが使えない場合の代替**。スキャン画面から呼び出すサブ画面。登録済み生徒の検索→イベント選択→手動打刻。完了後は必ずQRスキャン画面に戻る。 | GET /v1/students/search, POST /v1/attendance/manual |

- **主従関係**: 受付のメイン導線は **QRスキャン（/staff/scan）**。手動打刻は例外フローであり、完了後は **QRスキャン画面に戻る** ことを徹底する（§0 参照）。
- **出席確認（参加者一覧）**: イベント単位のため、`/admin/events` 一覧から各イベントの「参加者」リンク（例: `/admin/events/[eventId]/participants`）として表示するか、ホームに「出席確認」として「イベントを選んでから参加者一覧へ」のショートカットを出してもよい。2-3 では「管理者のイベント一覧から参加者へ」で十分とする。

### 2.3 管理者のみ表示（管理者専用）

| 表示ラベル       | パス                   | 説明                         | 対応API・備考 |
|------------------|------------------------|------------------------------|---------------|
| イベント管理     | `/admin/events`       | イベントCRUD・QR・出席レポート | GET /v1/admin/events 等 |
| 生徒名簿管理     | `/admin/students`     | 生徒一覧・新規登録・編集・削除・CSV取込 | students list/create/update/delete/import |
| スタッフ管理     | `/admin/staffs`       | スタッフ一覧・招待・権限変更   | GET/POST/PUT/DELETE /v1/admin/staffs, invite |
| 参加申込一覧（全利用者） | `/admin/registrations` | 全利用者の申込一覧（任意）   | （必要に応じて） |
| レポート出力     | `/admin/reports` または イベント別 | 出席CSVダウンロード        | GET /v1/admin/reports/events/{id}/csv |
| お知らせ管理     | `/admin/news`         | お知らせCRUD                 | GET /v1/news, POST/PUT/DELETE admin/news |

- 利用者・スタッフには上記カードを**表示しない**（リンクも出さない）。管理者のみ表示する。

## 3. 権限マトリックスとの対応（ホーム出し分けの根拠）

SECURITY_DESIGN 2.2 の「API 別 権限マトリックス」に基づき、**その API を呼べる役割**にだけホームからメニューを出す。

| メニュー（機能）     | 利用者(1) | スタッフ(2) | 管理者(3) | 根拠（API 権限） |
|----------------------|------------|-------------|-----------|-------------------|
| マイQR               | ○          | ○           | ○         | GET /v1/users/me/qr：認証済み全員 |
| イベント一覧         | ○          | ○           | ○         | GET /v1/events：認証済み全員 |
| 打刻履歴             | ○          | ○           | ○         | GET /v1/users/attendance/history：本人のみ |
| 参加申込一覧（自分） | ○          | ○           | ○         | GET /v1/users/registrations：本人のみ |
| マイページ           | ○          | ○           | ○         | GET /v1/users/me：認証済み全員 |
| 打刻スキャン（メイン） | ×          | ○           | ○         | POST /v1/users/attendance：スタッフ以上。受付の主画面。 |
| 手動打刻（例外・代替） | ×          | ○           | ○         | GET /v1/students/search, POST /v1/attendance/manual：スタッフ以上。QRが使えない場合のサブ画面。 |
| イベント管理         | ×          | ×           | ○         | GET/POST/PUT/DELETE /v1/admin/events：管理者のみ |
| 生徒名簿管理         | ×          | ×           | ○         | /v1/admin/students：管理者のみ |
| スタッフ管理         | ×          | ×           | ○         | /v1/admin/staffs：管理者のみ |
| レポートCSV          | ×          | ×           | ○         | GET /v1/admin/reports/events/{id}/csv：管理者のみ |
| お知らせ管理         | ×          | ○           | ○         | POST/PUT/DELETE /v1/admin/news：スタッフ以上（一覧は全員） |

- 上表の「○」の役割に対してのみ、該当メニューカードをホームに表示する。

## 4. 画面構成・コンポーネント

### 4.1 レイアウト

- **パス**: `/home`（既存の `app/home/page.tsx` → `HomeClient`）
- **認証**: 未認証なら `/login` へリダイレクト（既存の `useEffect` のまま）。
- **構成**:
  1. **ヘッダー**: タイトル「QRコード打刻システム」、挨拶「ようこそ、{loginId}さん」、**役割バッジ**（利用者 / スタッフ / 管理者）、マイページリンク、ログアウトボタン。
  2. **メイン**: 「機能」見出しの下に、**役割に応じたメニューカード**をグリッド表示。

### 4.2 出し分け条件（実装）

- **共通メニュー**: 常に表示（マイQR、イベント一覧、打刻履歴、参加申込一覧、マイページ）。
- **スタッフ以上**: `isStaff === true`（= role_flag 2 または 3）のとき、「打刻スキャン」「手動打刻」を追加。お知らせ管理はスタッフも可なら追加。
- **管理者のみ**: `isAdmin === true`（= role_flag === 3）のとき、「イベント管理」「生徒名簿管理」「スタッフ管理」「レポート出力」「お知らせ管理」「参加申込一覧（全利用者）」を追加。
- **レポート出力**: 管理者のみ表示。利用者・スタッフには表示しない（または「管理者のみ利用できます」の説明のみ表示するかは UX で決定。SECURITY_DESIGN に合わせて**管理者のみカードを表示**することを推奨）。

### 4.3 コンポーネント・データ

- **HomeClient**（`app/home/page.tsx` から利用）:
  - `useAuth()` で `user`, `roleLabel`, `isAdmin`, `isStaff`, `logout` を取得。
  - メニュー項目を「共通」「スタッフ用」「管理者用」の配列または設定で持ち、`isStaff` / `isAdmin` に応じてフィルタして表示。
- **メニューカード**: 各項目を `Link`（Next.js）でラップし、`href` は上記パス表に準拠。見た目は既存の `border rounded-lg hover:border-indigo-500` 等で統一。

### 4.4 パス・ルートの統一

- 既存の `HomeClient` は `/my-qr`, `/user/events`, `/history`, `/user/registrations`, `/staff/scan`, `/admin/events`, `/admin/registrations`, `/admin/students`, `/admin/staffs`, `/admin/reports` 等を参照している。
- 2-1 では `/events`（イベント一覧）、`/registrations`（申込一覧）、`/staff/manual`（手動打刻）等の案がある。**実装時は既存ルートと 2-1 の表を突き合わせて一本化**し、本設計の「表示ラベル ⇔ パス」だけはこのドキュメントの表を正とする。

## 5. 受け入れ条件（2-3 完了の定義）

- ログイン済みの利用者（role_flag=1）で `/home` にアクセスすると、**利用者向けメニューのみ**（マイQR、イベント一覧、打刻履歴、参加申込一覧、マイページ）が表示され、スタッフ用・管理者用のカードは表示されない。
- ログイン済みのスタッフ（role_flag=2）で `/home` にアクセスすると、上記に加え**スタッフ用メニュー**（打刻スキャン、手動打刻）が表示され、管理者専用のカードは表示されない。
- ログイン済みの管理者（role_flag=3）で `/home` にアクセスすると、**全メニュー**（イベント管理、生徒名簿管理、スタッフ管理、レポート出力、お知らせ管理、参加申込一覧（全利用者）を含む）が表示される。
- ヘッダーに現在の役割が分かる**役割バッジ**（利用者 / スタッフ / 管理者）が表示される。
- 未認証で `/home` にアクセスした場合は `/login` にリダイレクトされる（既存挙動の維持）。
- 各メニューカードのリンク先パスは、上記「2. 役割別メニュー一覧」および「4.4 パス・ルートの統一」に従う。
