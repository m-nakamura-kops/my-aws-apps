# 結合テスト テスト仕様書（マスター版）

QRコード打刻システムの結合テストにおいて、全機能を網羅するマスター仕様書です。  
各項目に **操作手順**・**期待される画面表示（API応答）**・**確認すべきDBテーブル/カラム** を記載します。

---

## 共通前提

- **認証**: 要認証APIは `Authorization: Bearer <token>` をヘッダーに付与する（ログインで取得した token を使用）。
- **APIベースURL**: ローカルは `http://localhost:3001`、デプロイ環境はその環境のURL。
- **role_flag**: 1=利用者、2=スタッフ、3=管理者。
- **テストデータ**: 検証用テストユーザー（管理者・スタッフ・利用者）を事前に作成済みであること。

---

### 利用者登録テスト（チェックリスト 1.2.1〜1.2.3）用アカウント

| チェック | 内容 | 用意するもの | 手順 |
|----------|------|----------------|------|
| **1.2.1** | 自己登録（成功） | 新規メール（シードに含めない） | フォームで `it-reg-new@example.com`（任意の未使用メール可）、パスワード `TestPass12`、名前・電話番号を入力して登録 → **201 Created** を確認。 |
| **1.2.2** | バリデーション | アカウント不要 | メール形式不正（`@` なし）で送信。例: `invalidemail`、パスワード8文字以上 → **400** とエラーメッセージを確認。 |
| **1.2.3** | 重複登録 | 既存アカウント 1 件 | 事前に `npm run seed-test-users` を実行し、**it-dup@example.com** を DB に投入する。同じメールで再度フォームから登録 → **400**（または 409）とエラーメッセージを確認。 |

- **1.2.3 用シード**: `it-dup@example.com` は `database/seed-test-users.sql` および `backend/scripts/seed-test-users.js` に含まれており、`npm run seed-test-users` で作成される。
- **注意**: 現在の `POST /v1/users/register` は重複メール時に `ON DUPLICATE KEY UPDATE` で更新し 201 を返す実装のため、重複時 400/409 を期待する場合は API を「既存メールなら挿入せずエラーを返す」ように変更する必要がある。

## 1. ユーザー管理

### 1.1 認証（ログイン・利用者登録）

#### 1.1.1 利用者ログイン（成功）

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. `POST /v1/users/login` を実行する。<br>2. リクエストボディに `{"email": "テスト利用者のメール", "password": "設定したパスワード"}` を送信する。 |
| **期待される画面表示（API応答）** | HTTP 200。レスポンス body に `token`, `userId`, `userName`, `orgId`, `roleFlag` が含まれる。`token` は空でない。 |
| **確認すべきDBテーブル/カラム** | `users`: 該当 `email` のレコードが存在し、`role_flag` が 1（利用者）であること。`password` は SHA-256 ハッシュで保存されていること（照合はAPI側で実施）。 |

| 項目 | 内容 |
|------|------|

| **確認すべきDBテーブル/カラム** | `users`: 該当 `email` のレコードは変更されない。 |

## 5. 履歴

#### 5.1 打刻履歴取得

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. 利用者トークンで `GET /v1/users/attendance/history` を実行する（クエリで email を渡す場合は本人の email）。 |
| **期待される画面表示（API応答）** | HTTP 200。自分の打刻履歴一覧が返る。 |
| **確認すべきDBテーブル/カラム** | `attendance_logs`: 該当 `email` のレコードが履歴の元。件数・`in_time`, `out_time`, `event_id` が一致すること。`events` と結合してイベント名等が返ること。 |

#### 5.2 履歴の内容

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. 上記履歴APIのレスポンスを確認する。 |
| **期待される画面表示（API応答）** | 各履歴にイベント名・日時・入退場時刻・滞在時間等が含まれる。 |
| **確認すべきDBテーブル/カラム** | `attendance_logs`: `in_time`, `out_time`。`events`: `event_name`, `event_date`。滞在時間は `out_time - in_time` と一致すること。 |

#### 5.3 他人の履歴が取得できないこと

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. 利用者Aのトークンで、別の利用者Bの email を指定して `GET /v1/users/attendance/history?email=B` を呼ぶ（APIが email パラメータを受け取る場合）。 |
| **期待される画面表示（API応答）** | 本人のみ返す仕様の場合、Aの履歴のみ返るか、Bの履歴は返さない（403 または フィルタで本人のみ返す）。 |
| **確認すべきDBテーブル/カラム** | `attendance_logs`: BのレコードがそのままDBに存在するが、APIはA用にAの履歴のみ返すこと。 |

#### 5.4 スタッフ・管理者の履歴

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. スタッフまたは管理者トークンで `GET /v1/users/attendance/history` を実行する。 |
| **期待される画面表示（API応答）** | 仕様が「自分のみ」の場合、HTTP 200 で自分自身の履歴のみ返る。全員分を返す仕様でない限り、他人の履歴は含まれない。 |
| **確認すべきDBテーブル/カラム** | `attendance_logs`: スタッフ/管理者の `email` に紐づくレコードのみが返る元であること。 |

## 6. シナリオテスト

#### S1 利用者登録→ログイン→イベント一覧→申込→打刻→履歴確認

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. `POST /v1/users/register` で利用者を登録。<br>2. `POST /v1/users/login` でログインし token 取得。<br>3. `GET /v1/events` でイベント一覧取得。<br>4. `POST /v1/users/events/{eventId}/register` で申込。<br>5. `POST /v1/users/attendance` で打刻。<br>6. `GET /v1/users/attendance/history` で履歴を取得し、打刻が含まれることを確認。 |
| **期待される画面表示（API応答）** | 各ステップで上記各項目の期待応答が返る。最終的に履歴に今回の打刻が1件含まれる。 |
| **確認すべきDBテーブル/カラム** | `users`: 1件追加。`registrations`: 1件追加。`attendance_logs`: 1件追加。履歴APIがこの1件を返すこと。 |

#### S2 管理者でイベント作成→QR取得→参加者確認→出席レポート

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. 管理者でログイン。<br>2. `POST /v1/admin/events` でイベント作成。<br>3. `GET /v1/admin/events/{eventId}/qr` でQR取得。<br>4. 別の利用者が申込・打刻する。<br>5. `GET /v1/admin/events/{eventId}/participants` で参加者一覧。<br>6. `GET /v1/admin/events/{eventId}/attendance-report` で出席レポート。 |
| **期待される画面表示（API応答）** | 各ステップで成功。参加者一覧に利用者が含まれる。出席レポートに打刻が反映される。 |
| **確認すべきDBテーブル/カラム** | `events`: 1件追加。`registrations`: 1件。`attendance_logs`: 1件。participants と attendance-report がこれらのデータと一致すること。 |

#### S3 生徒CSV取込→スタッフ招待→権限変更→利用者に変更

| 項目 | 内容 |
|------|------|
| **操作手順** | 1. 管理者で CSV import で生徒登録。<br>2. `POST /v1/admin/invite` でスタッフ招待。<br>3. `PUT /v1/admin/staffs/{email}` で権限変更。<br>4. `DELETE /v1/admin/staffs/{email}` でスタッフを利用者に変更。<br>5. `GET /v1/admin/staffs` で一覧から消えていることを確認。 |
| **期待される画面表示（API応答）** | 各ステップで成功。最終的にスタッフ一覧に該当ユーザーが含まれず、users では role_flag=1 で残っている。 |
| **確認すべきDBテーブル/カラム** | `users`: CSVで追加されたユーザー。招待で role_flag=2 または 3。権限変更で role_flag 更新。削除で role_flag=1 に更新、レコードは残る。スタッフ一覧は role_flag IN (2,3) のみのため、該当ユーザーは一覧に表示されないこと。 |
