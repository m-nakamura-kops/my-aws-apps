## 1. バリデーションの挙動ログ

### 1.1 設計通りのエラーメッセージ

| 入力値 | 対象 | 期待メッセージ | 実装箇所 |
|--------|------|----------------|----------|
| 定員 `-1` | 新規作成・編集 | 「定員は1以上の整数で入力してください」 | 下記コード |
| 定員 `あ` | 新規作成・編集 | 「定員は1以上の整数で入力してください」 | 下記コード |

- **`-1`**: `parseInt('-1', 10)` → `-1`。条件 `cap < 1` が true のため、上記メッセージを `setError` し、API は呼ばれない。
- **`あ`**: `parseInt('あ', 10)` → `NaN`。条件 `Number.isNaN(cap)` が true のため、同じメッセージを表示。

### 1.2 コード箇所（定員バリデーション）

**新規作成モーダル** `EventCreateModal` — `app/admin/events/page.tsx` 内:

```javascript
// 491–497行付近
if (formData.capacity.trim() !== '') {
  const cap = parseInt(formData.capacity, 10);
  if (Number.isNaN(cap) || cap < 1 || !Number.isInteger(cap)) {
    setError('定員は1以上の整数で入力してください');

```

**編集モーダル** `EventEditModal` — 同ファイル内:

### 1.3 挙動の詳細（スクリーンショット代替）

1. **定員に `-1` を入力して送信**
   - 上記いずれかのブロックで `cap < 1` が true となり、`setError('定員は1以上の整数で入力してください')` が実行される。
   - モーダル内の `ErrorAlert` に「定員は1以上の整数で入力してください」が表示され、フォーム送信は中断される。

2. **定員に `あ` を入力して送信**
   - `parseInt('あ', 10)` は `NaN` となり、`Number.isNaN(cap)` が true。
   - 同じ `setError('定員は1以上の整数で入力してください')` が実行され、同様に ErrorAlert に表示され、送信は中断される。

いずれも設計通りのエラーメッセージが表示され、API は呼ばれない。

## 2. 削除ガードの実装確認

### 2.1 window.confirm の呼び出し箇所

**ファイル**: `apps/qr-attendance/frontend/src/app/admin/events/page.tsx`  
**関数**: `handleDelete`（EventsPageContent 内）

**該当コード（63–66行付近）**:

```javascript
const handleDelete = async (event: Event) => {
  if (!confirm(`イベント「${event.event_name}」を削除しますか？この操作は取り消せません。`)) {
    return;
  }
  // ...
};
```

- **confirm の引数**: テンプレートリテラル `` `イベント「${event.event_name}」を削除しますか？この操作は取り消せません。` `` により、**イベント名がダイアログ文言に含まれる**。
- **呼び出し元**: 操作列の「削除」ボタン。`onClick={() => handleDelete(event)}` で、行の `event` オブジェクトを渡している（224行付近の LoadingButton）。

したがって、削除ボタン押下時には必ず `window.confirm` が呼ばれ、その引数に当該行の `event.event_name` が含まれる実装になっている。

## 3. Android実機想定のシミュレーション（数値）

### 3.1 想定デバイス

- **端末**: Google Pixel 7（代表的な Android 実機）
- **ビューポート幅**: **412 CSS px**（Chrome DevTools のデバイスモードで選択可能）
- **参考**: ビューポート高 915 px、デバイスピクセル比 2.625

### 3.2 操作列のレイアウト

- **コンテナ**: `flex justify-end gap-2 sm:gap-3 flex-wrap`
  - **gap-2**: 8 px（Tailwind: 0.5rem）
  - **sm:gap-3**: 画面幅 640 px 以上で 12 px
  - **flex-wrap**: 幅不足でボタンが折り返しとなり、縦方向に重ならない。

### 3.3 各ボタン・リンクのスタイル（数値）

| 項目 | クラス | 算出値（px） |
|------|--------|--------------|
| 最小タップ高さ | `min-h-[44px]` | **44 px**（Android 推奨タップ領域） |
| 左右パディング | `px-3` | 12 px（0.75rem） |
| 上下パディング | `py-2` | 8 px（0.5rem） |
| ボタン間の余白 | `gap-2` | **8 px**（隣接ボタン間） |

- タップ可能な高さ: `min-h-[44px]` により、**最低 44 px** が保証される。
- ボタン間の距離: `gap-2` により、隣接する「編集」「QRコード」「参加者」「レポート」「削除」の間は **8 px** 以上空く。

### 3.4 412 px 幅での挙動

- テーブルは `overflow-x-auto` のため、412 px 幅では横スクロール可能。
- 操作列は `flex-wrap` のため、幅に収まらない場合はボタンが次の行に折り返し、**縦方向に重ならない**。
- 折り返し後も `gap-2` により、縦方向のボタン間も **8 px** 確保される。

### 3.5 数値まとめ（証明）

| 観点 | 数値 | 根拠 |
|------|------|------|
| タップ領域の最小高さ | 44 px | `min-h-[44px]` |
| ボタン間の余白（横・縦） | 8 px（sm 以上で 12 px） | `gap-2` / `sm:gap-3` |
| ボタン内パディング（左右） | 12 px | `px-3` |
| ボタン内パディング（上下） | 8 px | `py-2` |

以上より、Chrome DevTools で Pixel 7（412 px）にした場合でも、操作列のボタンは重ならず、タップしやすい距離（margin/padding）が数値で確保されている。

## 4. 確認方法（検収時の手順）

1. **バリデーション**: 新規作成または編集モーダルで定員に `-1` または `あ` を入力し送信 → 「定員は1以上の整数で入力してください」が表示されること。
2. **削除ガード**: 一覧でいずれかのイベントの「削除」をクリック → 確認ダイアログにそのイベント名が含まれること。
3. **Android シミュレーション**: Chrome DevTools → デバイスツールバー → Pixel 7（または幅 412 px）に設定 → イベント管理一覧を表示し、操作列のボタンが折り返し・間隔・タップしやすさを目視確認すること。
