# 6.4.13・6.4.14・6.4.15 管理機能 詳細仕様案

**参照**: [デバイス・マトリクス完全版とハイブリッドUI設計指針](FE_DEVICE_MATRIX_AND_HYBRID_UI.md)  
**設計上の絶対遵守事項**: デバイス・マトリクス（基本PC/タブ・緊急時のみスマホ）、ハイブリッドUI（768px でテーブル↔カード、スマホ時は EmergencyMobileBanner）、検証可能性（実装完了時に客観的エビデンスを提示可能な設計）。

---

## 共通：デバイス・マトリクスとハイブリッドUI

以下の3画面はいずれも**管理者専用**であり、次のルールを適用する。

| 項目 | 内容 |
|------|------|
| **メイン端末** | PC・タブレット。通常業務は大画面で行う。 |
| **緊急時** | スマホ。出先などで最低限の確認・操作ができること。 |
| **画面幅 ≥768px** | 一覧は**テーブル形式**。警告バナーは**非表示**。 |
| **画面幅 <768px** | 一覧は**カード形式**。画面上部に **EmergencyMobileBanner**（「※現在は緊急用モバイル表示です」）を表示。 |
| **ブレークポイント** | Tailwind `md`（768px）を境界とする。`md:hidden` / `hidden md:block` で出し分け。 |
| **破壊的操作** | 削除・権限剥奪などは、モバイル表示時のみ**二段階確認**を検討する。 |

**検証可能性**: 実装完了時に、次の形式で客観的エビデンスを提示できるようにする。

- 0除算・空データ時の表示ロジックの該当コード抜粋とデバッグログ（該当する場合）
- 一覧の「先頭行のデータ形式」や画面テキストの再現（テキストベース）
- 767px / 768px 時のクラス適用状態（`md:hidden` 等）の明示

## 1. No. 6.4.13（管理者：生徒名簿管理）

### 1.1 概要

全生徒の一覧表示、氏名・メール等による検索、詳細情報の編集（氏名・カナ・区分・ステータス等）。  
**追加検討項目**: 『入会日』および『ステータス（有効/退会）』を管理項目に含める。

### 1.2 機能一覧

| 機能 | 内容 |
|------|------|
| **一覧表示** | role_flag=1（利用者/生徒）のユーザーを一覧表示。ページネーションまたは「もっと見る」対応。 |
| **検索** | 氏名（漢字）・氏名（カナ）・メールアドレスによる部分一致検索。既存 API `search` パラメータを利用。 |
| **詳細・編集** | モーダルまたは別画面で、氏名（漢字）・氏名（カナ）・電話・区分（org_id 等）・備考を編集。 |
| **新規登録** | 既存の「新規登録」モーダルを維持。メール・パスワード・氏名・カナ・電話・備考。 |
| **CSV一括登録** | 既存の CSV インポートを維持。 |
| **削除** | 生徒レコードの削除（確認ダイアログ必須）。モバイル時は二段階確認を推奨。 |

### 1.3 管理項目（表示・編集）

| 項目 | 説明 | 備考 |
|------|------|------|
| メールアドレス | ログインID。一覧・詳細で表示。編集は不可（キー）とするか、要検討。 | 既存 |
| 氏名（漢字） | name_kanji。編集可。 | 既存 |
| 氏名（カナ） | name_kana。編集可。 | 既存 |
| 電話番号 | tel。編集可。 | 既存 |
| 区分 | org_id または「生徒/一般」等のラベル。編集可。 | 既存（org_id 等） |
| 備考 | remarks。編集可。 | 既存 |
| **入会日** | **追加検討**。`users.created_at` を「入会日」として表示する。編集は不可（システム設定日）とする。 | 既存カラムで対応可 |
| **ステータス（有効/退会）** | **追加検討**。例: `status` または `is_active`（有効=1, 退会=0）。退会時はログイン不可・一覧でフィルタ可能とする。 | DB 拡張が必要な場合は migration を用意 |

### 1.4 画面構成

- **URL**: `/admin/students`（既存）
- **ヘッダー**: タイトル「生徒名簿管理」、ホームに戻る、新規登録ボタン、CSV一括登録。
- **検索**: 検索ワード入力＋検索ボタン。検索結果を一覧に反映。
- **一覧**:
  - **≥768px**: テーブル。列例: 氏名・カナ・メール・電話・入会日・ステータス・操作（編集・削除）。
  - **<768px**: カード。1 生徒 1 カード。カード内に氏名・メール・入会日・ステータス・編集・削除を配置。EmergencyMobileBanner をページ上部に表示。
- **編集**: モーダルまたはインライン。氏名・カナ・電話・区分・備考・ステータス（追加時）を編集。保存・キャンセル。

### 1.5 API（既存・拡張）

| 用途 | メソッド | エンドポイント | 備考 |
|------|----------|----------------|------|
| 一覧・検索 | GET | /v1/admin/students?limit=&offset=&search= | 既存。入会日は created_at、ステータスは拡張時追加。 |
| 新規作成 | POST | /v1/admin/students | 既存 |
| 更新 | PUT | /v1/admin/students/:email | 既存。ステータス項目を追加する場合は API 拡張。 |
| 削除 | DELETE | /v1/admin/students/:email | 既存 |
| CSV一括 | POST | /v1/admin/students/import | 既存 |

### 1.6 検証用エビデンス（設計時点で想定）

- 検索結果 0 件時の表示（エラーにならないこと、空一覧メッセージのテキスト再現）。
- 入会日・ステータス表示の該当コード箇所と、一覧の先頭行データ形式（テキスト再現）。
- 768px 境界でのテーブル/カード・バナー表示切り替えの Tailwind クラス適用箇所の提示。

## 2. No. 6.4.14（管理者：スタッフ管理）

### 2.1 概要

スタッフ一覧、新規スタッフの招待（メール送信等の導線）、権限（Admin/Staff）の変更・剥奪。

### 2.2 機能一覧

| 機能 | 内容 |
|------|------|
| **一覧表示** | role_flag IN (2, 3) のユーザーを一覧表示。スタッフ（2）と管理者（3）を区別表示。 |
| **検索** | 氏名・メール・カナによる部分一致検索。既存 API `search` パラメータを利用。 |
| **新規招待** | メールアドレスを指定してスタッフ/管理者を新規作成。初期パスワードを設定し、招待完了メッセージまたは「招待用URL・パスワードをコピー」等の導線を提供。※実際のメール送信は任意（現状「招待メールは送りません」でも可）。 |
| **権限変更** | スタッフ ↔ 管理者の切り替え（role_flag 2 ↔ 3）。確認ダイアログ必須。モバイル時は二段階確認を推奨。 |
| **権限剥奪** | スタッフ/管理者を「利用者」（role_flag=1）に変更。アカウントは残し、打刻記録は保持。確認ダイアログ必須。モバイル時は二段階確認を推奨。 |
| **編集** | 氏名・カナ・電話・備考の編集（任意）。 |

### 2.3 管理項目（表示）

| 項目 | 説明 |
|------|------|
| メールアドレス | 一覧・詳細で表示。 |
| 氏名（漢字・カナ） | 一覧・編集で表示。 |
| 権限 | Admin / Staff のラベル表示。変更ボタンで 2 ↔ 3 を切り替え。 |
| 登録日 | created_at。 |
| 打刻記録数 | 担当した打刻件数（任意）。既存 API で返している場合は表示。 |
| 操作 | 権限変更・利用者に変更（剥奪）・編集。 |

### 2.4 画面構成

- **URL**: `/admin/staffs`（既存）
- **ヘッダー**: タイトル「スタッフ管理」、ホームに戻る、**新規スタッフ招待**ボタン。
- **検索**: 検索ワード入力＋検索ボタン。
- **一覧**:
  - **≥768px**: テーブル。列例: 氏名・メール・権限・登録日・操作（権限変更・利用者に変更・編集）。
  - **<768px**: カード。1 人 1 カード。カード内に権限バッジ・操作ボタン。EmergencyMobileBanner をページ上部に表示。
- **招待モーダル**: メール・初期パスワード・権限（Staff/Admin）を入力。送信後は「招待しました。パスワードを伝えてください」等のメッセージ、またはクリップボードコピー導線。
- **権限変更**: スタッフ→管理者 / 管理者→スタッフ の確認ダイアログ。モバイル時は二段階確認。
- **利用者に変更**: 剥奪の確認ダイアログ（「このスタッフを利用者に変更しますか？アカウントは残り、打刻記録も保持されます」）。モバイル時は二段階確認。

### 2.5 API（既存・拡張）

| 用途 | メソッド | エンドポイント | 備考 |
|------|----------|----------------|------|
| 一覧・検索 | GET | /v1/admin/staffs?limit=&offset=&search= | 既存 |
| 招待（新規作成） | POST | /v1/admin/invite | 既存。body: email, password, role_flag 等。 |
| 権限変更 | PUT | /v1/admin/staffs/:email | 既存。body: role_flag。 |
| 利用者に変更（剥奪） | DELETE | /v1/admin/staffs/:email | 既存。role_flag=1 に更新。 |

### 2.6 検証用エビデンス（設計時点で想定）

- 招待完了後の表示内容（テキスト再現）と、権限変更・剥奪の確認文言の該当コード箇所。
- 一覧 0 件時の表示。768px 境界でのテーブル/カード・バナー表示のクラス適用箇所の提示。

## 3. No. 6.4.15（管理者：参加申込一覧）

### 3.1 概要

特定のイベントに閉じず、**全イベント**の申込状況を**時系列**で横断的に確認できる画面。

### 3.2 機能一覧

| 機能 | 内容 |
|------|------|
| **横断一覧** | 全利用者の全イベントに対する申込を 1 つの一覧で表示。イベント単位ではなく「申込」単位で行を並べる。 |
| **時系列** | デフォルトは**申込日時（created_at）の降順**。またはイベント開催日順の選択肢を用意してもよい。 |
| **フィルタ（任意）** | イベントID・イベント名・申込者メール・日付範囲で絞り込み。API が対応する場合はクエリで渡す。 |
| **ページネーション** | limit/offset または「もっと見る」で大量データに対応。 |
| **イベント・参加者への導線** | 各行から「当該イベントの参加者一覧」へリンク（既存 `/admin/events/:eventId/participants`）。 |

### 3.3 表示項目（1 行 = 1 申込）

| 項目 | 説明 |
|------|------|
| 申込日時 | registrations.created_at。時系列のキー。 |
| イベント名 | イベント名。リンクで参加者一覧へ。 |
| イベント開催日時 | event_date。 |
| 開催場所 | location（任意）。 |
| 申込者氏名 | name_kanji。 |
| 申込者メール | email。 |
| 操作 | 「参加者一覧」リンク（当該イベントの participants へ）。 |

### 3.4 画面構成

- **URL**: `/admin/registrations`（既存を拡張）
- **ヘッダー**: タイトル「参加申込一覧（全利用者）」、ホームに戻る。任意でフィルタ（イベント・日付範囲・申込者）の UI。
- **一覧**:
  - **≥768px**: テーブル。列例: 申込日時・イベント名・開催日時・申込者・メール・操作。
  - **<768px**: カード。1 申込 1 カード。カード内に申込日時・イベント名・申込者・「参加者一覧」リンク。EmergencyMobileBanner をページ上部に表示。
- **0 件時**: 「申込はまだありません」等のメッセージとイベント管理へのリンク。

### 3.5 API（新規または拡張）

現状の `GET /v1/users/registrations` は **email 必須**（自分の申込のみ）のため、**管理者用**に次のいずれかを満たす必要がある。

| 方式 | 内容 |
|------|------|
| **A. 新規エンドポイント** | `GET /v1/admin/registrations?limit=&offset=&event_id=&email=&date_from=&date_to=` を追加。管理者権限で全申込を返す。ORDER BY r.created_at DESC。 |
| **B. 既存の拡張** | `GET /v1/users/registrations` に管理者の場合は email を省略可能にし、省略時は全件返す。推奨は A（責務分離）。 |

レスポンス形式は既存の registrations 配列と揃える（reg_id, email, user_name, event_id, event_name, event_date, location, capacity, registration_date）。pagination 付き。

### 3.6 検証用エビデンス（設計時点で想定）

- 申込 0 件時の表示テキスト再現。フィルタ適用時の先頭行データ形式（テキスト再現）。
- 768px 境界でのテーブル/カード・バナー表示のクラス適用箇所の提示。時系列順の並び順を保証する API/コードの該当箇所。

## 4. 実装時のチェックリスト（共通）

各タスク完了時に、以下を満たすこと。

- [ ] 画面幅 ≥768px で一覧が**テーブル**、<768px で**カード**＋**EmergencyMobileBanner** が表示されている。
- [ ] 破壊的操作（削除・権限剥奪等）は、モバイル表示時は**二段階確認**を実装している（または仕様で不要と判断した理由を記載）。
- [ ] 空データ・0 件時にエラーにならず、適切なメッセージが表示される。
- [ ] VERIFICATION_REPORT と同様の形式で、**具体的なデータ**（画面テキスト再現・先頭行データ形式・クラス適用状態）に基づく検証報告書を提示できる。
