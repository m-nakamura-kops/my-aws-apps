## 0. 業務フロー・運用方針（PM確定）

以下の運用を絶対方針とする。

### 0.1 主従関係（QR優先思想）

| 方針 | 内容 |
|------|------|
| **デフォルトはQR打刻** | 本システムの**デフォルト（主軸）は QRコード打刻**である。受付の通常フローは利用者QRをスタッフがスキャンする方式とする。 |
| **手動打刻は例外フロー** | **手動打刻（/staff/manual）は、QRが利用できない場合の『代替措置（レスキュー・モード）』に過ぎない。** 手動打刻をメインとする運用は認めない。 |
| **受付のメイン画面** | **受付担当者のメイン画面は /staff/scan（QRスキャン）** とする。手動打刻画面は**スキャン画面から呼び出されるサブ画面**という位置づけとする。 |
| **完了後の強制リセット** | 手動打刻が完了した際は、確認メッセージを出した後、**デフォルトで『QRスキャン画面（/staff/scan）に戻る』ボタンを強調**して配置する。**自動的に次の生徒検索へ進むような『手動打刻をメインとするループ』は認めない。** |

### 0.2 未登録者・手動打刻UI（従来方針）

| 方針 | 内容 |
|------|------|
| **未登録者の扱い** | **原則、スタッフ権限での「その場での新規登録」は許可しない。** スタッフは登録済み生徒の検索・打刻のみ行う。 |
| **管理者の役割** | 未登録者が来た場合は、**管理者（Role: 3）** が **『生徒名簿管理』（生徒管理画面）** から新規登録を行った後、打刻処理を行う運用とする。 |
| **手動打刻画面のUI（スタッフ）** | **スタッフ（Role: 2）**: 登録済み生徒の**検索と打刻のみ**可能。新規登録ボタン・リンクは表示しない。 |
| **手動打刻画面のUI（管理者）** | **管理者（Role: 3）**: 必要に応じて **『新規登録画面』（生徒名簿管理）へのリンク** を配置し、登録後に手動打刻へ戻れるようにする。 |

- 生徒名簿管理 = `/admin/students`（生徒一覧・追加・編集・削除・CSVインポート）。管理者のみアクセス可。
- 新規登録画面へのリンクは「名前で見つからない場合はこちら」等の文言で、手動打刻画面から管理者のみに表示する。

## 3. 画面遷移の要点

- **未認証**: `/` → `/login`（または `/register`）。
- **認証後**: `/` または `/login` 成功 → `/home`。  
  `/home` は役割に応じて「利用者メニュー」「スタッフメニュー」「管理者メニュー」を出し分け。
- **利用者**: ホーム → イベント一覧 → イベント詳細（申込/取消）／マイQR／打刻履歴／申込一覧。
- **スタッフの受付導線（QR優先）**: ホーム → **QRスキャン打刻（/staff/scan）がメイン**。QRが使えない場合のみ、スキャン画面から **手動打刻（/staff/manual）** へ。手動打刻完了後は **必ずQRスキャン画面に戻る**（「QRスキャン画面に戻る」ボタン強調）。手動打刻をメインとするループは認めない。必要に応じて出席確認（参加者一覧）へ。
- **管理者**: 上記に加え、管理トップ → イベントCRUD・**生徒名簿管理（新規登録）**・スタッフ・レポート・お知らせ。未登録者が来た場合は生徒名簿で登録 → 手動打刻（またはQRスキャン）で打刻。手動打刻画面には「新規登録はこちら」リンクを表示し、登録後に戻れるようにする。**手動打刻完了後は同様にQRスキャン画面に戻る導線を徹底する。**

## 5. 手動打刻画面のUI要件（2-1 で確定する内容）

- **位置づけ（QR優先・再掲）**
  - 手動打刻は **QRが利用できない場合の例外フロー（レスキュー・モード）**。**メインの受付導線はQRスキャン（/staff/scan）**。
  - **入口**: 原則 **/staff/scan（QRスキャン画面）から「QRが使えない場合 → 手動打刻へ」リンクで遷移**。ホームからのリンクは代替入口として可。
  - **完了後の挙動（強制リセット）**: 打刻完了後は **確認メッセージを表示し、デフォルトで「QRスキャン画面（/staff/scan）に戻る」ボタンを強調して配置**する。**自動的に次の生徒検索へ進むような『手動打刻をメインとするループ』は認めない。**

- **入力**
  - イベント選択: ドロップダウンまたは一覧（GET /v1/admin/events または既存のイベント一覧API）
  - 生徒検索: テキスト入力 → 検索（GET /v1/students/search?q=...）、最小2文字程度推奨（登録済み生徒のみヒット）
- **表示**
  - 検索結果: 表またはカードで `name_kanji` / `name_kana` / `email` を表示、行クリックで「この人を打刻」に進む
  - 打刻実行: 選択生徒＋選択イベントで POST /v1/attendance/manual、成功時 **「打刻しました」等の確認メッセージ** を表示し、**目立つ「QRスキャン画面に戻る」ボタン**を表示。409 時「既に打刻済みです」表示。
  - **スタッフ（Role: 2）**: 上記のみ。新規登録の入口は表示しない。検索でヒットしない場合は「該当する生徒がいません。名簿にない場合は管理者にご依頼ください」等のメッセージを表示。
  - **管理者（Role: 3）**: 上記に加え、**「名前で見つからない場合 → 生徒名簿で新規登録」** のようなリンクを配置し、`/admin/students`（生徒名簿管理）へ遷移させる。登録後は手動打刻画面へ戻れるようにする（リンク先を `?returnTo=/staff/manual` 等で指定するか、案内文で「登録後はブラウザバックで手動打刻に戻れます」等を記載）。**いずれの場合も打刻完了後は「QRスキャン画面に戻る」を強調する。**
- **権限**
  - スタッフ・管理者のみ表示。利用者は 403 のため表示しない（またはルートでリダイレクト）。
- **運用（再掲）**
  - その場での新規登録はスタッフには許可しない。未登録者の対応は管理者が生徒名簿管理で行い、打刻は手動打刻画面で行う。**手動打刻はあくまで例外フローであり、完了後は必ずQRスキャン画面に戻す。**
