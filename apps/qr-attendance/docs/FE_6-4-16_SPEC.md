# 6.4.16 管理者：お知らせ管理 詳細仕様案

**参照**: [デバイス・マトリクス完全版とハイブリッドUI設計指針](FE_DEVICE_MATRIX_AND_HYBRID_UI.md)  
**前提**: 6.4.13〜6.4.15 は検証完了により**完了**とする。本仕様は 6.4 系の最終タスクである。

---

## 1. 概要

利用者ホーム（6.3）に表示する**重要なお知らせ**の作成・編集・削除を管理者が行う機能。  
お知らせは**公開/非公開**と**掲載期間（開始日・終了日）**を持ち、**公開かつ現在時刻が掲載期間内**のものだけを利用者ホームに新着順で表示する。

---

## 2. 追加仕様（承認時組み込み）

- **お知らせのタイプ分け**: 「重要（緊急）」と「通常」の2種類を選択可能にする。重要の場合は利用者画面で**赤色のバッジや枠線**で目立たせる。
- **未来予約の厳密性**: 掲載開始日が未来の場合、1秒でも前であれば利用者画面（GET /v1/news）に**一切流れない**よう、バックエンドのクエリ条件を徹底する（`published_at <= NOW()` を厳密に適用）。

---

## 3. 管理機能

### 3.1 機能一覧

| 機能 | 内容 |
|------|------|
| **一覧表示** | 全お知らせを一覧表示（公開・非公開・期間外を含む）。管理者が状態を把握できるようにする。 |
| **新規作成** | タイトル・本文・公開スイッチ・掲載開始日・掲載終了日を入力して作成。 |
| **編集** | 既存お知らせのタイトル・本文・公開スイッチ・掲載開始日・掲載終了日を変更。 |
| **削除** | お知らせを削除。確認ダイアログ必須。モバイル表示時は二段階確認を推奨。 |

### 3.2 管理項目（データ項目）

| 項目 | 説明 | 必須 | 備考 |
|------|------|------|------|
| **タイトル** | お知らせの見出し。 | ○ | プレーンテキスト。最大長は DB に合わせる（例: 255 文字）。 |
| **本文** | お知らせの詳細内容。 | ○ | **プレーンテキスト**のみ。改行は可。HTML はサニタイズして表示するか、そのままテキストとして表示。 |
| **公開/非公開スイッチ** | 掲載するかどうか。 | ○ | 「公開」= 掲載する、「非公開」= 利用者ホームに表示しない。 |
| **掲載開始日** | 表示を開始する日時。 | ○ | この日時以降にのみ表示される。 |
| **掲載終了日** | 表示を終了する日時。 | 任意 | NULL の場合は「無期限」（終了日なし）。 |

---

## 4. 表示ロジック（利用者ホーム）

### 4.1 表示条件

利用者ホームに表示するお知らせは、次の**3条件をすべて満たす**ものに限定する。

| 条件 | 説明 |
|------|------|
| **公開** | 公開/非公開スイッチが「公開」であること。 |
| **掲載開始後** | 現在時刻 ≥ 掲載開始日であること。 |
| **掲載終了前** | 掲載終了日が NULL であるか、現在時刻 ≤ 掲載終了日であること。 |

論理式で表すと:

- `is_published = 1`  
  AND `published_at <= NOW()`  
  AND `(expired_at IS NULL OR expired_at >= NOW())`

### 4.2 並び順・件数

- **並び順**: **新着順**（掲載開始日の降順、または作成日時の降順。仕様上は「新着順」のため `created_at DESC` または `published_at DESC` で統一）。
- **件数**: 全件表示するか、上限を設けるかは実装で決定（例: 最大 10 件まで表示など）。

### 4.3 利用者ホームでの表示仕様

- **表示タイミング**: 上記条件を満たすお知らせが**1件以上ある場合のみ**、ホーム画面の**上部**（ヘッダーとメニューカードの間）にブロックを表示する。
- **表示様式**: **目立つ形**でリスト表示する（例: 枠線・背景色・アイコンを使ったセクション「重要なお知らせ」）。
- **0件のとき**: お知らせブロックは**表示しない**（何も描画しない）。

## 5. データベース・API

### 5.1 既存テーブル `news` の利用と拡張

既存の `news` テーブルを利用する。

| カラム | 型 | 説明 |
|--------|------|------|
| id | INT PK | お知らせID |
| title | VARCHAR(255) | タイトル |
| content | TEXT | 本文（プレーンテキスト） |
| published_at | DATETIME | 掲載開始日時 |
| expired_at | DATETIME NULL | 掲載終了日時（NULL=無期限） |
| created_at | DATETIME | 作成日時 |
| updated_at | DATETIME | 更新日時 |

**拡張**: 公開/非公開を制御するため **is_published**（TINYINT 0/1、デフォルト 1）、種別のため **announcement_type**（TINYINT 1=通常・2=重要、デフォルト 1）を追加するマイグレーションを用意する。  
既存環境にカラムが無い場合は `COALESCE(is_published, 1)` および `COALESCE(announcement_type, 1)` で互換する。

### 5.2 管理者用 API（一覧・作成・更新・削除）

| 用途 | メソッド | エンドポイント | 備考 |
|------|----------|----------------|------|
| 一覧（管理用） | GET | /v1/admin/news?limit=&offset= | 全件。公開・非公開・期間外を区別して返す。管理者権限。 |
| 新規作成 | POST | /v1/admin/news | body: title, content, is_published, published_at, expired_at。 |
| 更新 | PUT | /v1/admin/news/:id | body: title, content, is_published, published_at, expired_at。 |
| 削除 | DELETE | /v1/admin/news/:id | 管理者権限。 |

管理用一覧のレスポンスには、表示条件の判定に必要な項目（is_published, published_at, expired_at, created_at）を含める。

### 5.3 利用者ホーム用 API（公開・期間内のみ）

| 用途 | メソッド | エンドポイント | 備考 |
|------|----------|----------------|------|
| 一覧（公開用） | GET | /v1/news?limit=&page= | **表示条件**: is_published=1 AND published_at<=NOW() AND (expired_at IS NULL OR expired_at>=NOW())。並び: 新着順（created_at DESC または published_at DESC）。認証済みユーザーのみ。 |

既存の `GET /v1/news` が上記表示条件と新着順を満たすよう拡張する（is_published を条件に含める）。

## 6. 画面構成

### 6.1 管理画面（/admin/announcements）

- **URL**: `/admin/announcements`（新規ページ。既存メニューが `/admin/news` の場合は、本機能では `/admin/announcements` に統一し、メニューリンクをここに変更する。）
- **権限**: 管理者のみ（RoleGuard: Admin）。
- **ヘッダー**: タイトル「お知らせ管理」、ホームに戻る（/home）、**新規作成**ボタン。
- **一覧**:
  - **≥768px**: **テーブル**。列例: タイトル・公開/非公開・掲載開始日・掲載終了日・作成日時・操作（編集・削除）。
  - **<768px**: **カード**。1 件 1 カード。カード内にタイトル・公開/非公開・掲載期間・操作。画面上部に **EmergencyMobileBanner** を表示。
- **新規作成・編集**: モーダルまたは別画面。入力項目: タイトル・本文（textarea）・公開/非公開スイッチ（チェックボックスまたはトグル）・掲載開始日（日時）・掲載終了日（日時、任意）。保存・キャンセル。
- **削除**: 確認ダイアログで「このお知らせを削除しますか？」等。モバイル時は二段階確認を推奨。

### 6.2 ハイブリッドUIの徹底

| 項目 | 内容 |
|------|------|
| **ブレークポイント** | 768px（Tailwind `md`）。 |
| **≥768px** | 一覧は**テーブル**（`hidden md:block`）。警告バナー**非表示**。 |
| **<768px** | 一覧は**カード**（`md:hidden`）。画面上部に **EmergencyMobileBanner**（「※現在は緊急用モバイル表示です」）を表示。 |
| **破壊的操作** | 削除は確認必須。モバイル時は二段階確認を推奨。 |

### 6.3 利用者ホーム（6.3）への組み込み

- **配置**: ヘッダー（タイトル・挨拶・ログアウト等）の直下、メニューカード（「機能」セクション）の**上**に「重要なお知らせ」ブロックを配置する。
- **表示条件**: 公開かつ掲載期間内のお知らせが **1件以上** ある場合のみブロックを描画する。
- **見た目**: 目立つ形（例: セクション見出し「重要なお知らせ」、リスト形式、日付またはタイトルのみ／クリックで本文展開など）。プレーンテキストのため、改行はそのまま表示する。

## 7. 検証・提出義務（VERIFICATION_REPORT.md）

### 7.1 掲載期間外のお知らせが表示されないことの証明

完了報告時には、**VERIFICATION_REPORT.md** を更新し、次のいずれか（または両方）で客観的エビデンスを提示する。

| 証明方法 | 内容 |
|----------|------|
| **A. API レスポンス** | 利用者ホーム用 API（GET /v1/news）を呼んだときのレスポンスに、**掲載期間外のレコードが含まれていない**ことを示す。例: (1) 掲載終了日が過去のお知らせを 1 件作成し、同一条件で GET /v1/news を実行したレスポンス JSON を掲載し、その id が含まれないことを明記する。(2) 掲載開始日が未来のお知らせを 1 件作成し、同様にレスポンスに含まれないことを示す。 |
| **B. 画面スクリーンショット＋条件** | (1) 掲載期間内のお知らせのみを 2 件作成し、利用者ホームで 2 件表示される画面をキャプチャする。(2) 1 件を「掲載終了日を昨日の日付」に編集して保存し、利用者ホームを再読み込みしたところ、その 1 件が**表示されなくなった**ことをキャプチャまたはテキストで再現する。 |
| **C. コード・クエリの明示** | 利用者向け一覧取得の SQL または API 内の条件式を抜粋し、「`is_published = 1` かつ `published_at <= NOW()` かつ `(expired_at IS NULL OR expired_at >= NOW())` のみ取得している」ことを示す。 |

最低限、**C** に加え **A** または **B** のいずれかで「掲載期間外が表示されない」ことを示す。

### 7.2 その他の検証エビデンス

- **ハイブリッドUI**: 767px / 768px でのテーブル・カード・バナー切り替えに使った Tailwind クラス（`hidden md:block` / `md:hidden`）の適用箇所を VERIFICATION_REPORT に記載する。
- **0件時**: 利用者ホームで、公開かつ期間内のお知らせが 0 件のとき、お知らせブロックが表示されないことをテキストまたはキャプチャで記載する。

## 8. 実装時のチェックリスト

- [ ] 管理画面の URL を `/admin/announcements` とし、768px でテーブル/カード切り替え・スマホ時は EmergencyMobileBanner を表示している。
- [ ] お知らせの作成・編集で、タイトル・本文（プレーンテキスト）・公開/非公開・掲載開始日・掲載終了日を設定できる。
- [ ] 利用者ホーム用 API が「公開 AND 掲載期間内」のみを返し、新着順で並んでいる。
- [ ] 利用者ホームで、該当お知らせが 1件以上ある場合のみ上部に「重要なお知らせ」を目立つ形で表示し、0件のときはブロックを出さない。
- [ ] 掲載期間外のお知らせが表示されないことを、上記 6.1 のいずれかの方法で証明し、VERIFICATION_REPORT.md に記載した。
- [ ] 削除は確認ダイアログ必須。モバイル時は二段階確認を推奨。
